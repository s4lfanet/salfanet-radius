## Configuration Overview

These files are the **authoritative source** for the FreeRADIUS configuration used by Salfanet RADIUS.
They are kept **in sync with the running production VPS** — all files verified April 15, 2026.

## Files Included

### Core Config Files

| File | VPS Path | Description |
|------|----------|-------------|
| `clients.conf` | `/etc/freeradius/3.0/clients.conf` | RADIUS client definitions (localhost + dynamic from DB) |
| `clients.d/nas-from-db.conf` | `/etc/freeradius/3.0/clients.d/nas-from-db.conf` | **Auto-generated by app** — NAS entries synced from DB every 5 min via `syncNasClients()` cron. Template only in this repo. |
| `mods-available/sql` | `/etc/freeradius/3.0/mods-available/sql` | MySQL module config (driver, credentials, pool) |
| `mods-available/rest` | `/etc/freeradius/3.0/mods-available/rest` | REST module — HTTP calls to Next.js app for authorize/post-auth/accounting |
| `mods-available/mschap` | `/etc/freeradius/3.0/mods-available/mschap` | MS-CHAP auth module (PPPoE) |
| `sites-available/default` | `/etc/freeradius/3.0/sites-available/default` | Main RADIUS virtual server (authorize, authenticate, accounting, post-auth) |
| `sites-available/coa` | `/etc/freeradius/3.0/sites-available/coa` | CoA / Disconnect-Request server (port 3799) |
| `policy.d/filter` | `/etc/freeradius/3.0/policy.d/filter` | Username/password filter policies (modified to allow `@rw02` format) |

## VPS Active File Structure

On VPS, the **enabled** modules and sites are:

```
mods-enabled/
  sql          → standalone FILE (not symlink) — copy of mods-available/sql
  rest         → symlink → /etc/freeradius/3.0/mods-available/rest
  + many default mods (chap, pap, mschap, files, etc. — symlinks from installer)

sites-enabled/
  default      → standalone FILE (not symlink) — copy of sites-available/default
  coa          → symlink → /etc/freeradius/3.0/sites-available/coa
  inner-tunnel → symlink → ../sites-available/inner-tunnel (enabled by default installer, harmless)
```

> **IMPORTANT for fresh install:** `mods-enabled/sql` and `sites-enabled/default` are **standalone files**
> on VPS (not symlinks). When deploying to a new VPS, **copy the file** — don't create a symlink for these two.
> All others (rest, coa) use symlinks.

## Fresh Install on a New VPS

### Step 1: Install FreeRADIUS

```bash
apt-get update && apt-get install -y freeradius freeradius-mysql freeradius-rest
```

### Step 2: Copy Config Files

```bash
cp clients.conf /etc/freeradius/3.0/clients.conf
cp clients.d/nas-from-db.conf /etc/freeradius/3.0/clients.d/nas-from-db.conf
cp mods-available/sql /etc/freeradius/3.0/mods-available/sql
cp mods-available/rest /etc/freeradius/3.0/mods-available/rest
cp mods-available/mschap /etc/freeradius/3.0/mods-available/mschap
cp sites-available/default /etc/freeradius/3.0/sites-available/default
cp sites-available/coa /etc/freeradius/3.0/sites-available/coa
cp policy.d/filter /etc/freeradius/3.0/policy.d/filter
```

### Step 3: Enable Modules & Sites

```bash
cd /etc/freeradius/3.0

# sql: copy as standalone file (NOT symlink -- matches production VPS)
cp mods-available/sql mods-enabled/sql
chown freerad:freerad mods-enabled/sql

# rest: symlink with absolute path (matches production VPS)
ln -sf /etc/freeradius/3.0/mods-available/rest mods-enabled/rest
chown -h root:root mods-enabled/rest

# sites: default as standalone file
cp sites-available/default sites-enabled/default
chown freerad:freerad sites-enabled/default

# coa: symlink
ln -sf /etc/freeradius/3.0/sites-available/coa sites-enabled/coa
```

### Step 4: MS-CHAP Fix (Ubuntu 22.04 + OpenSSL 3.0)

Required for PPPoE MS-CHAPv2 to work on Ubuntu 22.04+:

```bash
# Edit /etc/ssl/openssl.cnf
# In [provider_sect], add:
#   legacy = legacy_sect
# Add new section:
#   [legacy_sect]
#   activate = 1
```

See `openssl.cnf.example` in this repo for reference.

### Step 5: Set Permissions

```bash
chown -R freerad:freerad /etc/freeradius/3.0/
chmod 640 /etc/freeradius/3.0/mods-enabled/sql
chmod 640 /etc/freeradius/3.0/sites-enabled/default
chmod 640 /etc/freeradius/3.0/clients.d/nas-from-db.conf
```

### Step 6: Test & Start

```bash
freeradius -CX          # test config (stop service first)
systemctl enable freeradius
systemctl start freeradius
systemctl status freeradius
```

## Database Configuration

`mods-available/sql` credentials (update for new VPS if DB credentials differ):

```
login     = "salfanet_user"
password  = "salfanetradius123"
radius_db = "salfanet_radius"
server    = "localhost"
port      = 3306
```

**`read_clients = no`** — NAS clients are NOT read from the `nas` DB table.
Instead, the app's `syncNasClients()` cron writes them to `clients.d/nas-from-db.conf`
every 5 minutes (part of `freeradiusHealthCheck()`). FreeRADIUS is reloaded with
`systemctl reload freeradius` (SIGHUP) only when the file changes — active sessions
are NOT dropped.

## REST Module (mods-available/rest)

Calls the Next.js app at `http://localhost:3000`:

| Section | URL | Purpose |
|---------|-----|---------|
| `authorize` | `/api/radius/authorize` | Check user status BEFORE auth — returns Reject if expired/blocked |
| `post-auth` | `/api/radius/post-auth` | After successful auth — set `firstLoginAt`, `expiresAt`, create billing transaction |
| `accounting` | `/api/radius/accounting` | Every Start/Stop/Interim-Update — session tracking via radacct |

Pool: `start = 0`, `min = 0` — connections are made **lazily**
(Next.js app may not be running yet at FreeRADIUS startup).

## Auth-Type MS-CHAP

FreeRADIUS 3.x requires **lowercase** Auth-Type name:

```
Auth-Type mschap {   # correct (lowercase)
    mschap
}
# NOT: Auth-Type MS-CHAP { ... }  <- wrong, will fail silently
```

## CoA Configuration (Port 3799)

Listens for CoA/Disconnect-Request from the app:

```bash
# Test disconnect:
echo "User-Name = testuser" | radclient -x 127.0.0.1:3799 disconnect testing123

# MikroTik must have:
# /radius incoming set accept=yes
# RADIUS entry address must match SOURCE IP of CoA packet
# If NAT/masquerade on path, add extra RADIUS entry for the masqueraded source IP
```

## Username Format

`policy.d/filter` is modified to allow `@rw02`, `@rw03` format (no dot required after `@`).
This supports area-suffixed PPPoE usernames like `lilissusilawati@rw02`.
